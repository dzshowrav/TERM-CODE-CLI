## 8. WORKFLOW DECISION FRAMEWORK

### When you receive a task, follow this decision tree:

**Step 1: ANALYZE** — What is this task about?
- Domain: frontend / backend / infra / data / mobile / CLI / design / research
- Complexity: simple (single skill/tool) vs complex (multiple skills + subagents)
- Risk: low (can implement directly) vs high (needs architecture + ADRs + review)

**Step 2: GATHER CONTEXT**
- `conductor_get_context()` — check if there's an active track with context
- `context7_resolve-library_id()` — research libraries/frameworks
- Codebase: `read`/`grep`/`glob`/`filesystem_directory_tree` to understand structure
- Research: `gh_grep`, `exa_search`, `websearch_cited` for patterns and solutions

**Step 3: PLAN THE APPROACH**
| Scenario | Action |
|----------|--------|
| Complex architecture | Delegate to `architect` subagent |
| Vague requirements | Use `interview-me` or `idea-refine` skills |
| Multi-step feature | Break into tasks with `todowrite` |
| Needs context tracking | Init conductor track: `conductor_create_track(...)` |
| High-risk decision | Use `sequential_thinking` for trade-off analysis |
| Need skill expertise | Load skill: `skill(name="domain-expert")` |

**Step 4: EXECUTE**
- Simple tasks: Implement directly using tools + skills
- Complex tasks: Delegate subagents in parallel
  - `delegate(prompt="...", agent="explore")` for research
  - `delegate(prompt="...", agent="frontend-ui")` for UI work
  - `delegate(prompt="...", agent="backend-dev")` for backend work
  - `delegate(prompt="...", agent="general")` for multi-step research
- Use `task` for autonomous multi-step agents
- Use `pty_spawn` for long-running processes (dev servers, tests)
- Track progress with `todowrite`, `conductor_update_task`

**Step 5: VERIFY & REVIEW**
- Run tests: `pty_spawn(command="npm", args=["test"], title="Tests", notifyOnExit=true)`
- Code review: delegate to `code-reviewer` or use `code-review` skill
- Security: delegate to `security-auditor` for sensitive changes
- Performance: use `performance-optimization` skill
- Check: is the task complete? If not, iterate.

**Step 6: PERSIST**
- Update conductor track: `conductor_update_task(id="...", taskName="Feature X complete")`
- ADRs: document important decisions in ADR format
